A very good point, and one that I missed when writing the article originally. I recommend using EF entities as mementos for the actual domain entities. This puts the domain entity in the domain layer, but the EF entity in the persistence layer. They have two different concerns. Yet in the next breath, I recommend using lambda expressions to represent specifications. EF requires a lambda written in the persistence layer, but a specification should be in the domain layer. I'm mixing concerns. Looking back at the code I've written in the 2 years since writing this post, I can see now that I haven't really been writing DDD specifications. I've just been using lambda expressions within my domain model to query the persistence layer. The most surprising thing about this -- besides the fact that I hadn't noticed the problem before -- is that it has worked just fine. The consumers of my domain layer have never actually provided a specification, but yet they have always had the methods they need to get their work done. So it turns out I'm still doing DDD wrong. I try to do my projects the DDD-way, I think that it's a good way to implement layered application with good architecture and support for unit tests. However, I always have the feeling of "crossing the river for the water". I see that most people don't focus on maintaining the domain as it should be but are instead using the database entities as the business entities. In most cases, this is totally valid but not always. I like to think that I am programming against the reality - the domain, and not against a relational database, even if these often are connected as 1-1. I have seen several blogs about implementations for linq-queries against the repositories, IQueryable or not. All of these implementaions means that you are in fact writing lambda-expression against a database context and database entities (partial classes or not). The repositories are interfaces, as should be, but they will not really be exchangeable. With linq-queries in the domain that uses data entities you are kind of stuck with that data layer. Is this still DDD or is this an acceptable trade off? It seems this is an old post but I need to ask a question anyway.
